aweskill roadmap / SkillNexus notes / 0511

Borrow the quality loop, not the whole studio.

SkillNexus suggests a useful lifecycle: manage, run, capture history, evaluate, improve, and rank. For aweskill, the right adaptation is a set of small CLI surfaces that keep the local store and multi-agent projection model intact.

Design Position

aweskill should stay a local, CLI-first skill package manager. The next useful layer is evidence: can a skill run, what happened when it ran, how well did it perform, and what should be improved?

Keep Local store, source lifecycle, bundles, projection, and repair remain the product center.
Add Runnable, measurable, maintainable skill workflows with JSON output for agents and CI.
Avoid Hidden activation state, broad automatic rewrites, Electron-only interfaces, and service-first design.
Sequence Readiness before run, run before history, history before eval, eval before improve.

Implementation Order

  1. doctor status --json
  2. run <skill>
  3. execution log + history
  4. SKILL.tests.yaml + eval
  5. testcase from-history
  6. improve --out
  7. score list/top/stale
  8. MCP / JSON-RPC wrapper
Priority0

Readiness First

Add a read-only readiness probe before introducing evaluation or evolution. Agents need one stable answer to whether the store and projection state are usable.

Command Shape

  • aweskill doctor status --global --agent codex
  • aweskill doctor status --project . --agent codex
  • aweskill doctor status --global --agent codex --json

Reuse

  • Store initialization checks
  • agent list classification
  • doctor clean hygiene rules

Boundary

No --apply, no repair behavior, no second diagnostic model. Output readiness, counts, and suggested next commands.

Priority1

Run Skills Directly

SkillNexus has a Quick Run path. aweskill needs the CLI equivalent before history, eval, and improvement can become meaningful.

Command Shape

  • aweskill run <skill> --input "..."
  • aweskill run <skill> --stdin
  • aweskill run <skill> --json

Behavior

  • Resolve central-store skill names first
  • Read SKILL.md
  • Send skill instructions plus input to the configured provider
  • Print plain output by default

Open Question

Decide whether provider configuration belongs in ~/.aweskill/config.json, environment variables only, or a small provider subcommand.

Priority2

Execution History

Production usage is the strongest signal. Append-only logs can turn real runs into reviewable evidence and later candidate test cases.

Storage

~/.aweskill/logs/execution.ndjson

skill input output model durationMs caller success

Commands

  • aweskill history <skill>
  • aweskill history <skill> --json
  • aweskill run <skill> --no-log

Safety

Logs may contain sensitive user input. Document that clearly, provide opt-out behavior, and consider retention settings before adding destructive clear commands.

Priority3

Lightweight Eval

The most important SkillNexus idea is making skills measurable. Start with companion test files and explainable scoring.

Command Shape

  • aweskill eval <skill>
  • aweskill eval <skill> --case SKILL.tests.yaml
  • aweskill eval <skill> --json

Judge Types

  • grep for expected text
  • command for deterministic checks
  • llm only when configured

Dimensions

correctness instruction following safety completeness robustness executability cost awareness maintainability
Priority4

Improve Without Overwriting

Evolution is valuable, but automatic rewriting is risky in a package manager. Generate candidates, explanations, and diff-friendly output first.

Command Shape

  • aweskill improve <skill> --from-eval latest --out <dir>
  • aweskill improve <skill> --from-history --out <dir>
  • aweskill improve <skill> --json

Inputs

  • Latest eval failures
  • Repeated history failures
  • Existing SKILL.md

Guardrail

Never overwrite the managed store by default. Preserve frontmatter and linked assets unless a change is justified and visible.

Priority5

Local Score and Trending

SkillNexus Trending acts like an asset map. aweskill can expose the same signal as a CLI maintenance view over local eval, history, and source metadata.

Command Shape

  • aweskill score list
  • aweskill score list --json
  • aweskill score top --by maintainability
  • aweskill score stale

Signals

  • Latest total score
  • Failing dimensions
  • Last evaluated and last used
  • Source update availability
Priority6

MCP/API After CLI Stability

JSON-RPC or MCP can be useful, but only after command semantics settle. The service layer should wrap the existing library behavior, not create parallel rules.

Candidate Methods

  • skill.run
  • skill.eval
  • skill.history
  • store.list
  • agent.status

Rule

MCP/API should be an adapter over tested CLI/library semantics.

Not Now

  • Do not build an Electron desktop studio.
  • Do not start with a complex multi-engine evolution framework.
  • Do not add a hidden global activation registry.
  • Do not overwrite generated or evolved skills by default.

Human Reading Lens

Each priority should answer one question: what evidence do we need before the next capability is safe and useful?
