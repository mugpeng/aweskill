SkillClaw study notes · 2026-05-11

SkillClaw 给 aweskill 的启发

SkillClaw 的重点不是安装 Skill，而是让 Skill 从真实会话中积累、验证、同步和演化。 aweskill 不应该照搬它的代理服务器形态，但可以学习它的生命周期思想，把本地包管理推进到质量管理、候选审核、清晰同步和可读报告。

核心判断

aweskill 的优势是 CLI-first、本地中央仓库、Bundle、跨 Agent 投影和 Doctor 修复。 SkillClaw 的优势是使用反馈、共享存储、候选验证、Dashboard 和演化循环。最合理的学习方式是吸收后者的质量闭环，但保留 aweskill 的本地、显式、可审计边界。

aweskill 应保持

一个中央 store，一个清楚的投影模型，一套能被 Agent 和 CI 调用的 CLI。

aweskill 可学习

质量元数据、staged review、manifest sync、HTML report，以及可选的离线演化插件。

install/import validate staged review approve project run/eval/history score/report optional improve

学习顺序

  1. 先读 aweskill 的设计边界
  2. 再读 SkillClaw 的 SkillManager
  3. 继续看 SkillHub 的 manifest 和 rollback
  4. 把 validation 和 dashboard 当作产品方向
  5. 最后才考虑 evolve 或 proxy
Lesson 1

Skill 是有状态资产

一个 Skill 不只是目录和 `SKILL.md`。它还应该有 review、质量、风险、来源、验证状态和维护建议。

Lesson 2

候选不应直接激活

外部导入、LLM 生成、自动演化产生的 Skill 应先进入 staged 区，通过检查和人工确认后再进入 store。

Lesson 3

共享需要 manifest

团队同步不能只是复制文件夹。manifest、tree hash、版本和备份能让 sync 行为可解释、可回滚。

Priority0

质量元数据

先让 aweskill 知道每个 Skill 的质量状态，但不要修改 `SKILL.md` 本身。

  • 新增 `~/.aweskill/metadata/skills/<skill>.json`。
  • 记录 rating、notes、knownIssues、lastReviewedAt、validationStatus。
  • 提供 `aweskill store quality list/show/set`，并从第一版支持 `--json`。
explicit metadata no telemetry agent friendly JSON
Priority1

显式验证

SkillClaw 有候选验证思想。aweskill 可以先做确定性的本地验证，而不是一上来引入 LLM judge。

  • `aweskill store validate` 检查 `SKILL.md`、frontmatter、目录名、引用文件、脚本和正文。
  • 输出分为 error、warning、info。
  • 验证只报告，不默认修复；修复仍属于 doctor 命令。
Priority2

Staged 候选区

新 Skill 先进入候选区，避免未审查内容直接影响 Agent 行为。

  • 路径：`~/.aweskill/staged/<skill>/`。
  • 命令：`staged list/show/validate/approve/reject`。
  • staged Skill 不允许投影，approve 时必须检测中央仓库冲突。
Priority3

Manifest Sync

借鉴 SkillHub 的 manifest 和 hash 思想，先做本地目录同步，再考虑 Git、S3 或 OSS。

  • `aweskill store export ./team-skills`。
  • `aweskill store sync ./team-skills --dry-run`。
  • manifest 包含 name、description、version、treeSha256、files、source、updatedAt、status。
  • 默认 dry-run，apply 前备份，不默认 mirror-delete。
Priority4

先报告，后 Dashboard

SkillClaw 的 Dashboard 很有启发，但 aweskill 可以先做静态报告，把数据模型跑顺。

  • `aweskill doctor report --json` 给 Agent 和 CI 用。
  • `aweskill doctor report --html` 给人看。
  • 报告包含 store、bundles、agents、projection 分类、validation、staged、source update 和建议命令。
Priority5

可选 run/history/eval

如果未来要学习 SkillClaw 的真实使用反馈，aweskill 可以用显式 `run` 和本地 history 做更小的闭环。

  • `aweskill run <skill> --input "..."`。
  • `~/.aweskill/logs/execution.ndjson` 只在日志策略明确后启用。
  • `aweskill eval <skill>` 使用 `SKILL.tests.yaml`，支持 grep、command、可选 LLM judge。
Priority6

演化能力放到可选层

自动改写 Skill 很强，但也带来隐私、冲突、模型调用和质量风险。更稳妥的做法是插件化和 staged-first。

  • `aweskill improve <skill> --from-eval latest --out <dir>`。
  • `aweskill evolve from-log ./session.md --staged`。
  • 生成结果默认进入 staged，不能直接覆盖中央仓库。

暂时不要做

不要把本地代理、隐藏 telemetry、OSS/S3 团队同步、完整 Dashboard、自动覆盖式 evolve 直接塞进核心。 这些能力都应该建立在质量元数据、验证、staged review、manifest sync 和 report 已稳定的基础上。

建议实现顺序

  1. `store quality`
  2. `store validate`
  3. `staged` 工作流
  4. `store export/sync`
  5. `doctor report`
  6. 可选 `run`
  7. history 和 eval
  8. 可选 improve/evolve 插件

关键边界

  • 中央 store 是唯一事实来源。
  • 投影文件系统状态就是启用状态。
  • 质量数据独立于 `SKILL.md`。
  • 所有危险操作先 dry-run 或 staged。
  • 生成内容必须可审查。
